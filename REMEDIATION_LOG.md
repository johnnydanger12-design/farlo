## Iteration 10 (continued) ‚Äî full non-sampled re-verification pass: CRITICAL finding ‚Äî `compile_user_data_export`/`delete_account_data` callable by anyone, unauthenticated

**Citation:** the operating prompt's explicit requirement that no category may be claimed A+ without "full (non-sampled) re-verification" first. This finding was surfaced by that re-verification, not by a targeted audit ‚Äî a concrete demonstration of why the requirement exists. Files: new migration `20260705224500_restrict_data_export_and_account_deletion_functions_to_service_role.sql`.

- **Relocate:** re-running `get_advisors(type: security)` as part of Backend/Supabase's re-verification surfaced 56 lints, 21 of which were `anon_security_definer_function_executable`/`authenticated_security_definer_function_executable` warnings across 21 distinct `SECURITY DEFINER` functions. Rather than wave these off as routine linter noise, pulled the full body of all 21 and categorized every one: 19 are safe regardless of the grant ‚Äî they either check `auth.uid()` against the relevant row internally (`auth_user_owns_truck`, `get_transfer_counterparty`, `set_owner_review_response`, `invite_employee_by_email`, etc. ‚Äî anon's `auth.uid()` is `NULL`, so these self-scope to nothing), are trigger-only functions not meaningfully invocable outside a real trigger context (`update_truck_rating`, `notify_truck_owner_on_review`, `restrict_orders_payment_status_column`, etc.), or are deliberately narrow, low-sensitivity public lookups already reviewed and accepted in an earlier iteration (`find_profile_by_email`, `profile_display_name`, `profile_stripe_connected`, `get_truck_follower_count`, `owner_has_active_subscription`). **Two were not:** `compile_user_data_export(p_user_id)` and `delete_account_data(p_user_id)` ‚Äî neither checks the caller's identity against the `p_user_id` argument at all, because both were built to be called exclusively by their Edge Functions using the service-role key, and it was never verified that `anon`/`authenticated` genuinely couldn't call them directly.
- **Confirmed live and exploitable, not just theoretical:** on the isolated `remediation` branch, inserted a synthetic victim user, then made a real unauthenticated HTTP request using only the branch's public `anon` key ‚Äî `POST /rest/v1/rpc/compile_user_data_export {"p_user_id": "<victim uuid>"}` ‚Äî with **zero** JWT, zero login, zero auth header beyond the anon API key that ships embedded in every client build. **It returned the victim's full profile, email, and account data.** This is a genuine, severe, unauthenticated PII-exfiltration vulnerability ‚Äî any caller who knows (or enumerates/guesses) a user's UUID can pull their complete data export. `delete_account_data` was also confirmed callable via the same unauthenticated path (`HTTP 204`, no error) ‚Äî its destructive `DELETE`/`UPDATE` statements happened not to take effect in this specific test (root cause not fully pinned down ‚Äî plausibly an RLS interaction specific to this test's table state), but relying on that as a safety net would be fragile and wrong; the function's own logic provides no protection at all, so the fix applies to both regardless of which one's impact was more fully realized in testing.
- **Fix:** `REVOKE EXECUTE ... FROM PUBLIC, anon, authenticated` plus `GRANT EXECUTE ... TO service_role` on both functions. **First attempt was incomplete** ‚Äî revoking from `PUBLIC` alone left both roles still genuinely able to execute (`has_function_privilege('anon', ...)` still returned `true` afterward), because `anon`/`authenticated` held separate, explicit, direct grants, not just an inherited `PUBLIC` grant. Caught this by re-checking `has_function_privilege()` after the first fix rather than assuming the first migration worked, and by re-running the *exact same live exploit request* ‚Äî it still succeeded. Revoking from `anon`/`authenticated` explicitly (in addition to `PUBLIC`) is what actually closed it.
- **Green:** re-ran the identical unauthenticated exploit request against the isolated branch post-fix ‚Äî `{"code":"42501", "message":"permission denied for function compile_user_data_export"}`, `HTTP 401`, for both functions. Confirmed the legitimate path is untouched: a direct privileged call to `compile_user_data_export` still returns real data (the Edge Function's service-role path is unaffected). Applied the identical fix to **production** immediately given the severity ‚Äî a live, currently-exploitable, unauthenticated PII-exfiltration path is not something to leave open while finishing the rest of a re-verification pass. Re-ran the exact same exploit request against **production itself** with the real public anon key: `HTTP 401`, `permission denied`, for both functions. Re-confirmed the legitimate GDPR export flow still works end-to-end post-fix by queuing one more real request against the `apple.review@farlo.app` demo account and letting the actual cron process it.
- **Honest gap:** did not fully root-cause *why* `delete_account_data`'s destructive statements didn't visibly execute in the one live exploit test (favorites row count unchanged after the anon call, despite `HTTP 204`) ‚Äî plausibly RLS interacting with the SECURITY DEFINER context in a way not fully traced, since time was prioritized on confirming and closing the definite, proven `compile_user_data_export` leak and applying the same fix to both functions regardless. This does not weaken the fix itself (the EXECUTE grant is revoked either way), but the exact mechanics of the near-miss are not fully understood.
