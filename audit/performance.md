# Farlo — Performance Audit (Phase 8: Discovery)

**Scope:** Flutter app (`lib/`, 116 Dart files, 26,793 lines) + a real `flutter build apk --analyze-size --target-platform android-arm64 --dart-define-from-file=.env.json` release build (succeeded, 188.3s Gradle task, full log at `/private/tmp/.../scratchpad/build_size.log`, JSON at `~/.flutter-devtools/apk-code-size-analysis_01.json`). Discovery only — no code changes, no commits. Read as a senior mobile performance engineer, calibrated against what actually ships in `build/app/outputs/flutter-apk/app-release.apk`, not assumptions.

**Grounding read in full before starting:** `audit/architecture.md` (Phase 1, god screens + 18 direct-Supabase files), `audit/supabase-audit.md` (Phase 2, 27 unindexed FKs, duplicate RLS policies re-evaluating `auth.uid()` per row), `audit/code-quality.md` (Phase 5, 19 non-`.autoDispose` family providers, 4 unbounded queries, `manage_hours_screen.dart` N+1), `audit/ux-review.md` (Phase 6, loading/error-state findings), `audit/security.md`/`audit/app-store-review.md` (iOS background location is architecturally broken — `NSLocationWhenInUseUsageDescription` only, no path to "Always" authorization, so `allowBackgroundLocationUpdates: true` in `location_tracking_service.dart:45` cannot function on iOS in practice). None of these are re-derived below; each is cited briefly for context, then taken one level deeper on the performance axis specifically.

## 1. Executive Summary

**Overall verdict: functionally acceptable at today's data volume (single/double-digit rows per table), but built with almost no performance instrumentation and several patterns that degrade non-linearly as usage grows.** The good news, confirmed by a real release build rather than guesswork: the compiled Dart app code itself is lean (`package:farlo` is only 976 KB of the 9 MB decompressed AOT payload) and the bundled image assets are small (3.9 MB). The bad news is structural: zero request timeouts anywhere in the networking layer, zero pagination (`.range()` used **0 times** in the entire codebase), a map screen that recomputes marker clustering on every single pan/zoom frame, 6 god screens with no widget-level rebuild scoping, and a truck-profile screen that fires 6 independent uncoordinated network round-trips on open. None of this is visible today because every table in production has single-digit-to-double-digit row counts (per Phase 2: `orders` 0 rows, `event_booking_requests` 1 row, `food_trucks` 9 rows) — these are latent-until-scale problems, not today's user complaints.

**Measured numbers, front and center:**
- **Release APK (arm64-only, `--analyze-size`): 41.3 MB built / 39 MB compressed.** `lib/arm64-v8a` (native code) is **21 MB** — over half the app — dominated by the Stripe, Firebase, Google Sign-In, and RevenueCat native SDKs, not Flutter/Dart code. `classes.dex`+`classes2.dex`+`classes3.dex` (Java/Kotlin bytecode from those same native SDKs) total **8 MB**. `resources.arsc` is **3 MB**. `assets/flutter_assets` (our own bundled images) is **4 MB**.
- **Bundled image assets: 3.9 MB total**, concentrated in 2 oversized PNGs: `assets/images/onboarding.png` (**1.8 MB**) and `assets/images/icon.png` (**1.6 MB**, used only as a launcher-icon/splash source, never displayed at full res in-app). Both are strong WebP/`pngquant` candidates — this is 87% of the entire asset budget in two files.
- **Icon-font tree-shaking is working correctly and is a non-issue**: `MaterialIcons-Regular.otf` shrank 1,645,184 → 14,744 bytes (99.1%), `Font-Awesome-7-Free-Solid-900.otf` 414,664 → 1,456 bytes (99.6%), `CupertinoIcons.ttf` 257,628 → 848 bytes (99.7%) in the release build log. Phase 5's flagged-unused `cupertino_icons` package costs **under 1 KB** in the shipped binary — worth removing for hygiene, not for size.
- **Zero `.range()` calls anywhere in `lib/`** (pagination is entirely absent) — Phase 5's "4 unbounded queries" finding is the visible tip of a codebase-wide pattern: of 67 total `.select(...)` call sites, only 5 cap results at all (via `.limit()`, no true offset paging), and 0 request timeouts exist anywhere except a single RevenueCat `.timeout(seconds: 10)` call in `main.dart:50`.
- **`truck_profile_screen.dart` fires 6 independent, uncoordinated network round-trips on open** (`foodTruckProvider`, `truckReviewsProvider`, `myReviewProvider`, `favoritedTruckIdsProvider`, `truckFollowerCountProvider`, `announcementPrefProvider`) — new finding, not previously cited by Phase 1/5's N+1 analysis, which looked at write loops, not this read-side fan-out.
- **Map screen recomputes full marker-clustering math on every map-camera-move frame** (`map_screen.dart:69-73`,`410-445`): `_mapController.mapEventStream.listen((_) { setState(() {}); })` triggers a full `MapScreen` rebuild — including `.where(_inVisibleBounds)`, `.sort()`, and `_applyClusterOffsets()` over the entire truck list — on every single frame of a pan/zoom gesture (potentially 60/sec during a drag), not debounced. This is the underlying mechanism behind Phase 6's live-observed stacked-pin bug — it's a genuine performance defect, not just a visual one.

**Top 5 highest-impact findings:**
1. **No network timeouts anywhere** (§3) — a hung Supabase call (bad cell signal, server hiccup) blocks the UI indefinitely with a spinner that never resolves; there is no `.timeout()` wrapping any Supabase query or Edge Function invocation in the entire app.
2. **Zero pagination, unbounded queries feeding eager `ListView(children:)`** (§5, cross-ref Phase 5 §2.15) — quantified further here: of ~15 total list-fetch call sites across the app, **13 have no `.limit()` and 0 have `.range()`** — pagination isn't a partial gap, it's absent as a concept in this codebase.
3. **Map screen's per-frame rebuild** (§2) — the highest-traffic screen in the app (default launch route) does expensive list filter/sort/cluster math synchronously inside `build()` on every pan/zoom frame with no memoization or debounce.
4. **19 non-`.autoDispose` family providers accumulate open Postgres Realtime channels for the app's lifetime** (Phase 5 §2.7, quantified for memory impact here in §9) — a consumer browsing 15-20 truck profiles in one session ends that session holding 15-20 permanently-open Realtime channels plus their retained provider state.
5. **`onboarding.png` (1.8 MB) and `icon.png` (1.6 MB) are 87% of the entire asset bundle** (§1/§6) — the single cheapest, lowest-risk fix in this whole report (recompress, no code change).

**Launch-readiness verdict:** None of these are the kind of bug that gets an App Store submission rejected (that's Phase 4's territory) or that a user will notice today at 9 food trucks and 0 orders in the database. But several of these (no timeouts, no pagination, per-frame map rebuilds, unbounded realtime channel growth) are exactly the shape of problem that turns into "the app feels slow" reviews the month usage crosses from test-data scale to real-city scale — and unlike a UI polish pass, several of these (pagination, timeouts) require touching both the repository layer and the consuming screens, so they get more expensive to retrofit the longer they're deferred.

## 2. Render Performance

- **Map screen: expensive, non-memoized computation runs inside `build()` on every map-move frame.** `map_screen.dart:69-73`: `_mapEventSub = _mapController.mapEventStream.listen((_) { if (mounted) setState(() {}); });` — `flutter_map`'s `mapEventStream` emits on every intermediate frame of a pan/zoom gesture, not just at gesture-end. Each emission triggers a full `MapScreen.build()`, inside which `map_screen.dart:410-445` does (inline, in the `MarkerLayer.markers` closure, recomputed from scratch every call): `.where(_inVisibleBounds)` filter over the full truck list, a `.sort()` by `sessionStartedAt`, and `_applyClusterOffsets(sorted)` (offset math to de-overlap nearby pins) — none of it cached against the previous frame's result, none of it gated by a debounce the way the *search* input already correctly is (`map_screen.dart:142`, 350ms `Timer`). **Severity: High.** **User-facing impact:** on a truck-dense map view, every pixel of drag/pinch re-runs O(n log n) sort + clustering math synchronously in the frame budget — at today's 9 trucks this is invisible, but it is precisely the mechanism Phase 6 caught live (3 pins rendering stacked with truncated badge text) — the clustering math is being re-run so often and so cheaply-implemented that it doesn't reliably converge to a stable non-overlapping layout even at low truck counts. **Recommended action:** move `_applyClusterOffsets`/`_inVisibleBounds`/sort into a memoized value recomputed only when the truck list or the *rounded* viewport bounds change (not every frame), and add the same debounce pattern already used for search (`map_screen.dart:142`) to the map-move listener before triggering a rebuild.
- **`ref.watch()` is called at the top of six 1,000+ line screen `build()` methods with no widget-level scoping** — `dashboard_screen.dart` (5 watches), `map_screen.dart` (5), `truck_profile_screen.dart` (12), `booking_requests_screen.dart` (10), `calendar_screen.dart` (6), `account_screen.dart` (6). Only **9 `Consumer(...)` usages exist app-wide** and **0 `Selector`/`.select((state) => ...)` narrowing calls** — every watched provider change forces Riverpod to re-run the entire enclosing `build()`, which for `dashboard_screen.dart:372-679` is a documented 307-line method (Phase 5 §2.6). **Severity: Medium.** **User-facing impact:** an unrelated provider update (e.g. `cartProvider` changing while viewing `truck_profile_screen.dart`, which watches it once at `:1184` inside the same 1,426-line build tree as the 5 other truck-detail providers) forces Flutter to re-diff the whole screen's widget tree rather than a scoped subtree, adding avoidable frame work — not currently dropping frames at today's data size, but a real cost multiplier once list sizes/interaction frequency grow. **Recommended action:** wrap the specific widgets that actually need each provider's value in `Consumer`/extract them into small `ConsumerWidget`s, reserving the outer `build()` for static structure.
- **`const` adoption is roughly 50% in the god screens, a real missed-optimization, not just a style nit.** Sampled proxy counts: `dashboard_screen.dart` has 37 non-`const` vs. 36 `const` `Icon(Icons....)`/`SizedBox(height/width: ...)` instantiations; `truck_profile_screen.dart` has 8 non-`const` vs. 5 `const` `Icon(Icons....)` calls. `analysis_options.yaml` inherits only the default `flutter_lints/flutter.yaml` with all customization commented out — `prefer_const_constructors` is nominally part of that default set but `flutter analyze` surfaces **zero** const-related warnings in a 26,793-line codebase (confirmed: only 2 unrelated info-level lints total, cross-ref Phase 5 §1), meaning either the analyzer isn't catching these missed opportunities or the file-level style is inconsistent enough that the lint doesn't fire on them contextually (mixed const/non-const sibling widgets in the same list). **Severity: Low-Medium.** **User-facing impact:** each non-const static widget subtree (an `Icon`, a fixed-size `SizedBox`, a static `Text` label) is rebuilt and reallocated every time its parent rebuilds instead of being reused as a canonical instance — individually cheap, but multiplied across a 1,000+ line screen with 5-12 `ref.watch()` triggers per rebuild (previous bullet), this is a real cumulative GC-pressure/frame-time cost, not a hypothetical one. **Recommended action:** a mechanical `dart fix --apply` pass plus enabling `prefer_const_constructors`/`prefer_const_constructors_in_immutables`/`prefer_const_literals_to_create_immutables` explicitly in `analysis_options.yaml` (currently relying on defaults with no verification they're actually firing).
- **`ListView(children: [...])` (eager, non-lazy) is used in 15 places vs. `ListView.builder` in only 2** (`manage_menu_screen.dart`, `booking_chat_screen.dart`) and `ListView.separated` in 5 — this is Phase 5's finding (§2.15), re-confirmed here from the render-performance angle: every one of those 15 eager lists builds and lays out **every row** on every screen open, regardless of scroll position, which is real wasted `build()`/layout work independent of the underlying unbounded-query problem in §5 (fixing the query cap without also converting these to `.builder` still leaves all N rows built eagerly, just with a smaller N). **Severity: High** (compounds directly with §5). **Recommended action:** couple the pagination fix in §5 with a `ListView.builder`/`SliverList.builder` conversion at the same call sites — doing one without the other leaves half the win on the table.
- **`ClipRRect` used 16 times, `Opacity` 3 times, `BackdropFilter` 0 times** — no evidence of the classic Opacity-in-a-hot-list-item anti-pattern (`Opacity` widgets are not concentrated in map markers or list-item hot paths on inspection); `ClipRRect` usage is spread across ~10 files for standard rounded-corner image/card treatments, not stacked/nested in a way that would force extra offscreen compositing passes. **Severity: Low / clean finding** — this dimension is not a real problem in the current codebase.

## 3. API Latency

- **Zero request timeouts on any Supabase call or Edge Function invocation in the entire app.** Confirmed via `grep -rn "\.timeout(" lib` — the only hit in the whole codebase is `main.dart:50`, wrapping the RevenueCat `Purchases.configure(...)` call in a 10-second timeout (a defensive fix already in place for that one specific integration). Every other network call — every repository method's `.from(...).select(...)`, every `functions.invoke(...)` (20 call sites per Phase 5 §2.17), every Stripe `presentPaymentSheet()` call — has no client-side timeout and relies entirely on the underlying `http`/Supabase client's own (unconfigured, default) socket-level timeout behavior. **Severity: High.** **User-facing impact:** on a degraded connection (elevator, parking garage, rural signal — realistic conditions for a food-truck-finder app used outdoors), a hung request leaves the calling screen's spinner (one of 55 `CircularProgressIndicator` instances, §10) spinning indefinitely with no fallback, no retry prompt, and no way for the user to know whether to wait or restart the app. This is a more severe version of Phase 5's error-handling findings — those cover *failed* requests; this covers requests that *never resolve at all*. **Recommended action:** wrap the shared repository call pattern (ideally the same fire-and-forget wrapper Phase 5 §2.1 already flagged as duplicated 5× — a good place to also centralize a timeout) with a project-wide default (`Duration(seconds: 15-20)` is a reasonable starting point for a mobile client on Supabase's REST layer), surfaced as a distinct "request timed out, tap to retry" state rather than an indefinite spinner.
- **Zero retry/backoff logic anywhere** (`grep -rn -i "retry\|backoff" lib` returns only UI `Text('Retry')` labels/`onRetry` callback *parameters* on 3 screens — `subscription_screen.dart:38`, `favorites_screen.dart:39`, `truck_profile_screen.dart:48` — no actual retry *execution* logic, e.g. no automatic re-attempt on a transient 5xx/network blip). **Severity: Medium.** **User-facing impact:** every transient failure (a momentary network drop mid-request) becomes a user-visible error requiring a manual tap, rather than being silently absorbed by one or two automatic retries — a real UX/reliability tax on a mobile app used outdoors with variable connectivity, though not a correctness bug. **Recommended action:** a single retry-with-backoff wrapper (1-2 attempts, short exponential delay) around read-only `GET`-equivalent Supabase queries specifically (not mutations, to avoid double-write risk) would meaningfully reduce user-visible errors with low implementation risk.
- **`truck_profile_screen.dart` issues 6 independent, uncoordinated round-trips on screen open — a new query-fan-out finding, not previously cited.** `foodTruckProvider(truckId)` (`:40`, joined truck+hours+menu, 1 round trip), `truckReviewsProvider(truck.id)` (`:223`), `myReviewProvider(truck.id)` (`:224`), `favoritedTruckIdsProvider` (`:245,251,254`, watched 3× in the same build but backed by one provider instance so not 3 separate fetches), `truckFollowerCountProvider(truckId)` (`:707`), `announcementPrefProvider(truckId)` (`:1398`) — 5 distinct providers, each independently triggering its own fetch the first time it's watched, with no `Future.wait`/batching and no single "truck detail bundle" endpoint. **Severity: Medium.** **User-facing impact:** opening a truck profile fires 5 separate HTTP round-trips to Supabase's REST layer instead of 1-2 batched ones — each has its own connection/TLS-negotiation and RLS-policy-evaluation overhead (compounding Phase 2's finding that `auth.uid()` is re-evaluated per-row across nearly every RLS policy); on a slow connection this manifests as content "popping in" in 5 separate waves (reviews, follower count, announcement toggle state all arriving at different times) rather than one coherent load. **Recommended action:** either a single Postgres RPC/view that returns truck+reviews+follower-count+announcement-pref in one call, or at minimum group the independent ones with `Future.wait` at the provider layer so they fire concurrently rather than however Riverpod's build-order happens to schedule them (today they're concurrent-by-accident since each is a separate provider watched in the same frame, but there's no explicit coordination or shared loading state — the screen can't currently show one unified "loading truck..." state, only per-section spinners).
- **Dashboard screen has a similar, smaller-scale fan-out**: `ownerTruckProvider` (joined), `_stripeConnectedProvider` (`dashboard_screen.dart:30`, autoDispose), `_activeOrdersProvider` (`:44`, family by truckId, joined with `order_items`), `_profileDisplayNameProvider` (`:60`, family by userId) — 4 separate round-trips for one screen open, none batched. **Severity: Low-Medium** (smaller than the truck-profile case, and one of the four is legitimately `.autoDispose`, showing the pattern is at least partially applied here).
- **Map screen does not re-query the server on every camera move** — confirmed as a clean/positive finding. `activeTrucksProvider` (`map_provider.dart:11-13`) is a `StreamProvider` backed by a Realtime channel + one initial fetch (Phase 1 §7, Flow A); truck data is *not* refetched on pan/zoom, only the local marker layout (§2) is recomputed. The per-frame cost identified in §2 is a client-side rendering problem, not a server-round-trip problem — worth being precise about which one it is, since the fix is entirely different (memoization, not debounced network calls).
- **CORS note (cross-ref Phase 2 §5):** no Edge Function sets `Access-Control-Allow-Origin`; irrelevant to the mobile app's latency (native HTTP clients aren't subject to CORS) but would matter if `web/index.html`'s Flutter web build ever calls an Edge Function directly from browser JS — flagged in Phase 2, repeated here only because it's adjacent to "API latency" and worth not re-discovering later.

## 4. Caching

- **Image caching is inconsistently applied — `Image.network` (no cache) is the dominant pattern, `CachedNetworkImage` (has cache) is the exception, despite the package already being a dependency.** `cached_network_image: ^3.4.1` is declared in `pubspec.yaml:44` and actually used in only **2 files**: `account_screen.dart` (own avatar) and `review_card.dart` (reviewer avatars). Every truck-photo, menu-item-photo, and truck-logo rendering — the actual high-frequency, high-data-volume image content in the app — uses raw `Image.network(...)` instead: **17 call sites across 8 files** (`truck_map_pin.dart:40`, `map_screen.dart:560,610,862`, `truck_bottom_sheet.dart:43,319,392`, `truck_profile_screen.dart:633,678,996,1119`, `dashboard_screen.dart`, `manage_menu_screen.dart` ×2, `edit_truck_screen.dart` ×2, `favorites_screen.dart`). **Severity: High.** **User-facing impact:** `Image.network` does use Flutter's in-memory `ImageCache` (so the same URL won't re-decode within one app session), but it has **no disk cache** — every truck logo, menu photo, and photo-carousel image is re-downloaded from Supabase Storage from scratch on every cold app launch, even for trucks the user has viewed dozens of times before. For a consumer who reopens the app daily to check their 5 favorite trucks, this means re-downloading the same ~5-15 images every single session indefinitely — real, avoidable mobile data cost and load-time cost that `cached_network_image` (already a dependency, already correctly used for 2 minor cases) would eliminate for the other 17 sites with a near-mechanical find/replace. **Recommended action:** replace the 17 `Image.network` call sites with `CachedNetworkImage`, prioritizing `truck_map_pin.dart`/`map_screen.dart`/`truck_bottom_sheet.dart` first (highest view-frequency: every map pin, every session).
- **No `cacheWidth`/`cacheHeight`/`memCacheWidth`/`memCacheHeight` specified anywhere** (`grep -rn "cacheWidth\|cacheHeight\|memCacheWidth\|memCacheHeight" lib` → zero matches). **Severity: Medium**, cross-referenced with §6/§9. **User-facing impact:** every displayed image — including a 44×44 map pin thumbnail (`truck_map_pin.dart:40`) and a 100×100 profile logo (`truck_profile_screen.dart:678-681`) — decodes at its full source resolution into memory before Flutter's `BoxFit` scales it down for display. If an owner uploads a photo taken on a modern phone camera (commonly 3000×4000px+) as their truck logo, every consumer who sees that truck's map pin decodes and holds a full multi-megapixel bitmap in memory just to paint a 44×44 icon. See §9 for the memory-footprint quantification.
- **No HTTP response caching layer of any kind** — no `dio` (not a dependency), no interceptor, no ETags/`Cache-Control` handling visible anywhere in the `http`/Supabase client usage. Every provider re-fetches from Supabase on every fresh watch (subject to Riverpod's own provider-instance caching, see next bullet), with no additional HTTP-layer cache in front of it.
- **Provider-level staleness policy: providers cache for the lifetime of the provider instance, refetch only on explicit invalidation (realtime push) or on a fresh navigation that creates a new instance — no time-based staleness/TTL policy exists anywhere.** For non-family providers (e.g. `ownerTruckProvider`), Riverpod's default keep-alive-while-listened behavior means the data persists as long as a widget is watching it, and refetches are realtime-driven (`ref.invalidateSelf()` on Postgres change, `food_truck_provider.dart:34`) rather than time-based — this is a reasonable, deliberate pattern, not a bug. The problem (Phase 5 §2.7, referenced not re-litigated) is specifically the **family** providers, which create a new, never-torn-down cache entry per ID rather than a single cache with a staleness window — so this isn't "the app refetches too often," it's "the app never lets go of what it already fetched," a different and in some ways worse caching failure mode (§9 quantifies it).
- **No offline/local-first behavior at all — confirmed 100% online-only.** No `hive`/`sqflite`/`drift`/`isar` dependency exists (`grep -in "hive\|sqflite\|drift\|isar" pubspec.yaml lib` → zero real hits). `shared_preferences` is used in exactly 5 files (`theme_provider.dart`, `messaging_repository.dart`, `map_screen.dart` — recent-searches list, `onboarding_provider.dart`, `calendar_screen.dart`) purely for small UI-preference flags, not as a data cache. **Severity: Medium (design choice, not obviously wrong for this app's use case)** — a food-truck locator's core value proposition (live location, live open/closed status) is inherently online-only by nature, so full offline support would arguably be over-engineering; flagged here as a discovery note per the audit brief, not as a defect. **What happens on a dropped connection**: per Phase 5 §2.16, most catch blocks either show a raw `'Error: $e'` string or silently swallow the failure (`debugPrint`-only) — there is no graceful degraded-mode UI (e.g. "showing last-known truck locations, may be outdated") anywhere; a dropped connection produces either an error screen or a silently-stuck loading spinner (§10), never a stale-but-usable view of previously-fetched data.

## 5. Database Efficiency (client query-construction angle — Phase 2 covered server-side indexes/RLS)

- **`.select()` calls: 67 total call sites. 27 are bare `.select()` (fetches every column, Supabase's default), 1 is explicit `.select('*')`, and the remaining 39 specify a column list or an embedded join.** So **28 of 67 (42%) of all select calls fetch full rows** when a narrower projection would do — e.g. `bookings_repository.dart:99` (`.select('*')`) and the many bare `.select()` sites in `employees_repository.dart`, `reviews_repository.dart`, `notifications_repository.dart`, `map_repository.dart`. **Severity: Medium.** **User-facing impact:** over-fetching columns is invisible at today's row counts and column widths, but it's pure wasted payload — for wide rows (e.g. `food_trucks` with description/social-link/hours-adjacent columns), a bare `.select()` used only to check existence or grab one field (e.g. `map_repository.dart:14`, feeding `activeTrucksProvider`) pulls every column across the wire for every row, every time. **Recommended action:** default to explicit column lists everywhere except where every column genuinely is needed (e.g. the joined truck-detail fetch already correctly does this at `food_truck_repository.dart:13,22`) — the codebase already knows how to do this well in ~40% of cases, it's a rollout-completeness problem, not a knowledge gap.
- **Zero `.range()` calls anywhere in `lib/` — pagination as a concept does not exist in this codebase.** Only 5 call sites use `.limit()` at all (`map_repository.dart:65` → 10, `notifications_repository.dart:15` → 100, `employees_repository.dart:113` → 30, `employees_repository.dart:274` → 50, `shifts_provider.dart:27` → 1 — Phase 5 §2.10 already flagged these 5 inconsistent hardcoded values), and **none of the 5 pair a `.limit()` with a corresponding `.range()`/offset for a "load more" mechanic** — a `.limit()` alone just truncates the result set with no way for the UI to ever request page 2. Combined with Phase 5's 4 fully-unbounded queries (`orders_repository.dart:80,104`; `bookings_repository.dart:23,49`, zero limit at all), **the accurate picture is: of roughly 15 total list-returning repository methods identified across the app, 4 are fully unbounded and the other ~11 either cap at an arbitrary single-page limit with no more-loading path, or fetch a small enough table that it doesn't matter yet** (e.g. `operating_hours`/`menu_items`, naturally bounded to ≤7/≤dozens of rows per truck). **Severity: High** (restates and quantifies Phase 5 §2.15's finding with the added precision that this isn't a partial gap — pagination genuinely doesn't exist as an implemented pattern anywhere to extend). **Recommended action:** implement one canonical `.range(offset, offset+pageSize-1)` "load more"/infinite-scroll helper and apply it first to the 4 confirmed-unbounded queries (orders, booking requests) since those are the ones most likely to accumulate real row counts fastest once the app has real usage.
- **Large joins/embeds are used appropriately, not excessively** — spot-checked `food_truck_repository.dart:13,22` (`'*, operating_hours(*), menu_items(*)'`), `orders_repository.dart:83` (`'*, order_items(*), food_trucks(name)'`), `employees_repository.dart:62` (`'food_trucks(*, operating_hours(*), menu_items(*))'`) — these are legitimate one-round-trip joins replacing what would otherwise be N+1 client-side fetches, a genuinely good pattern where it's used. No case was found of an embed pulling in more nested data than the consuming screen actually renders. **Clean finding for this specific sub-dimension.**

## 6. Image Optimization

- **Bundled static assets: 3.9 MB total, 87% concentrated in 2 files.** `du -sh` + per-file breakdown: `assets/images/onboarding.png` **1.8 MB**, `assets/images/icon.png` **1.6 MB**, `assets/images/Farlo Logo.png` **496 KB** — only 3 files exist under `assets/images/` (the sole `assets:` entry in `pubspec.yaml`). **Severity: Medium.** **User-facing impact:** `onboarding.png` is displayed once, full-screen, on the very first screen a new user ever sees (`onboarding_screen.dart`, cross-ref Phase 6's finding that this screen is also the least design-system-integrated) — 1.8 MB for one static illustration is 3-5× larger than a well-compressed PNG or WebP of equivalent visual complexity should be. `icon.png` (1.6 MB) is used only as the *source* image for `flutter_launcher_icons`/`flutter_native_splash` codegen (`pubspec.yaml`'s `flutter_launcher_icons:`/`flutter_native_splash:` blocks both reference it) — **it should not need to ship inside `assets/flutter_assets` at runtime at all** if it's purely a codegen source; confirm whether it's actually referenced by any in-app `Image.asset('assets/images/icon.png')` call (a quick grep found none — `grep -rn "images/icon.png" lib` → no hits), meaning this 1.6 MB is very likely pure dead weight in the shipped bundle today. **Recommended action:** (1) recompress `onboarding.png` via WebP or `pngquant`/`oxipng` — illustration art with flat color regions typically compresses 60-80% smaller losslessly; (2) confirm `icon.png` is unreferenced by any runtime `Image.asset` call and, if so, either move it outside `assets/images/` (e.g. to a `design/` or tool-only source folder excluded from `pubspec.yaml`'s `assets:` list) or accept the current 1.6 MB cost consciously if launcher-icon regeneration convenience is preferred over bundle size.
- **User-uploaded truck/menu photos: client-side compression exists and is reasonable, but is quality-only, never resolution-capped.** `ImagePicker().pickImage(..., imageQuality: 85)` (`edit_truck_screen.dart:125`, truck logo), `imageQuality: 80` (`edit_truck_screen.dart:130`, truck photos; `manage_menu_screen.dart:344`, menu-item photos) — all three apply JPEG-quality compression on pick, a real and correct mitigation. **However, none of the three specifies `maxWidth`/`maxHeight`**, unlike the one place in the app that gets this fully right: `account_screen.dart:488-493` (avatar picker) sets `maxWidth: 512, maxHeight: 512, imageQuality: 85` — resolution-capped *and* quality-compressed. **Severity: Medium.** **User-facing impact:** an owner photographing their truck or a menu item on a modern phone (commonly 3000-4000px on the long edge) uploads a multi-megapixel image at 80-85% JPEG quality — quality compression alone on a 4000px source image still produces a multi-MB file, when the largest place that image is ever displayed in the app is a photo-carousel tile or a menu-item card, both well under 1000px wide (`truck_profile_screen.dart:633-636` carousel, `:996-999`/`:1119-1122` menu-item cards — none specify a display width above `double.infinity` inside a bounded-height container). Every consumer who views that truck's profile then downloads that same oversized original (§4 confirms no caching softens this on repeat views) — this is a real, compounding data-cost and load-time problem on both the upload and every subsequent download. **Recommended action:** add `maxWidth`/`maxHeight` (e.g. 1600px long-edge is generous for any in-app display size) to the 3 truck/menu `ImagePicker` calls, matching the pattern `account_screen.dart` already established correctly for avatars.
- **No CDN/image-transform layer is used — Supabase Storage's on-the-fly image transformation (resize/format-convert via URL query params, a real Supabase Storage feature) is not used anywhere** (`grep -rn "render/image\|width=\|resize=" lib` for Supabase transform-URL patterns → no hits; all Storage URLs are consumed as plain public-bucket URLs). **Severity: Medium.** **User-facing impact:** even if upload-side resolution capping (previous bullet) is fixed, every different display context (44×44 map pin vs. 100×100 profile logo vs. full-width carousel) currently fetches and decodes the exact same stored file — there's no server-side "give me the 44px thumbnail" request, so the map pin case in particular (highest view-frequency of any image in the app, per truck, per map load) always pulls whatever resolution was actually stored. **Recommended action:** given Supabase Storage's image transformation is available on this project's plan tier (should be confirmed operationally, not assumed), routing the map-pin and list-thumbnail `Image`/`CachedNetworkImage` widgets through a transform-URL variant (e.g. `?width=88&height=88&resize=cover`) would let the two other fixes in this section (upload-side cap + `cacheWidth`/`cacheHeight`, §4/§9) compound into a much larger win than either alone.

## 7. Cold Starts

`lib/main.dart` read in full (63 lines) — everything that runs before `runApp()`:
1. `WidgetsFlutterBinding.ensureInitialized()` (`:20`) — required, no cost concern.
2. **`if (_stripePublishableKey.isNotEmpty) { Stripe.publishableKey = ...; await Stripe.instance.applySettings(); }`** (`:22-25`) — awaited, blocking, before any other init step. Per Phase 6's live-testing note, this call is what actually crashes the app entirely on macOS desktop (`MissingPluginException`) since `flutter_stripe` doesn't support that platform — not an iOS/Android production concern, but confirms this call has real platform-coupling cost and is unconditionally in the synchronous cold-start path on the platforms that do support it.
3. Supabase-config fail-loud check (`:27-36`) — cheap, synchronous, a deliberate defensive fix per the in-code comment (previous App Store rejection from a silently-stripped `assert`). No performance cost.
4. **`await Supabase.initialize(url: ..., publishableKey: ...)`** (`:38-41`) — awaited, blocking.
5. **`if (Firebase.apps.isEmpty) { await Firebase.initializeApp(...); }`** (`:43-45`) — awaited, blocking.
6. **`await Purchases.configure(...).timeout(const Duration(seconds: 10))`** (`:48-56`) — awaited, but correctly `.timeout()`-wrapped and `try/catch`-guarded so a stuck RevenueCat init can add **at most 10 seconds** to cold start, not hang forever — the *one* place in the whole app with both a timeout and a documented rationale for it (`:53-55` comment). Still, in the pathological case, this is a real **10-second worst-case addition to time-to-first-frame**, since `runApp()` (`:58`) doesn't fire until after this step resolves or times out.
7. `runApp(const ProviderScope(child: AppShell()))` (`:58`).
8. `PushNotificationService.initialize()` (`:62`) — correctly **not** awaited, with an explicit comment (`:60-61`) noting `getToken()` hangs on simulator/no-APNs — good, deliberate practice, already non-blocking.

**Severity: Medium.** **Findings:**
- **Steps 2, 4, 5, 6 are four independent, unrelated-to-each-other async initializations run fully sequentially with `await`, not concurrently.** Stripe's SDK settings, Supabase's client, Firebase's app, and RevenueCat's SDK have no dependency on one another (RevenueCat's key selection depends on `Platform.isIOS`, not on any prior step's *result*). Today's cost is likely small (each of Stripe/Supabase/Firebase init is typically tens-to-low-hundreds of milliseconds on a modern device), but stacked sequentially rather than via `Future.wait([...])`, the total blocking time is the **sum** of all four rather than the **max** of the four. **User-facing impact:** adds an estimated (not measured — no profiling tool was run against a cold-launch trace in this pass) tens to a couple hundred milliseconds of avoidable time-to-first-frame delay on a typical device, up to the full 10-second RevenueCat timeout in a degraded-network worst case. **Recommended action:** run steps 4-6 (Supabase/Firebase/RevenueCat — leave Stripe's settings-apply first since a couple of later flows implicitly assume `Stripe.publishableKey` is set) concurrently via `Future.wait`, each still individually try/catch-guarded the way RevenueCat already is.
- **Native splash screen is configured, not left blank** — `flutter_native_splash: ^2.4.4` is a dev dependency with a real config block in `pubspec.yaml` (color `#FFFFFF`/`#121212` light/dark, `image: "assets/images/Farlo Logo.png"`, Android-12-specific `android_12:` block with its own colors) — this is the *correct* tool for avoiding a blank white frame before Flutter's first paint, not a manual/ad-hoc splash. **However, the Android 12 splash migration is mid-edit in the working tree right now** (`git status`: 10 deleted `android12splash.png` variants across every density bucket, `values-v31/styles.xml` and `values-night-v31/styles.xml` both modified, new untracked `mipmap-anydpi-v26/` and `colors.xml`) — this looks like an in-progress switch from a bespoke Android-12-splash-image approach to the platform's native adaptive-icon-based splash (`mipmap-anydpi-v26` is the standard adaptive-icon mechanism). **Severity: Medium, process risk not yet a defect** — if committed and built in its current partial state, a broken reference from `styles.xml` to a now-deleted `android12splash.png` drawable would reintroduce exactly the "blank/wrong frame before first paint" problem `flutter_native_splash` exists to prevent, specifically on Android 12+. **Recommended action:** before this phase's findings are acted on, first confirm (outside this audit's read-only scope) that the in-progress Android splash migration is completed and a real device/emulator cold-launch on an Android 12+ target shows the intended splash, not a placeholder or crash — this is a "verify before it ships" item, not a code fix this audit should make.
- **Initial route (`/map`, `router.dart:42`) eagerly builds `MapScreen`, which constructs a `flutter_map` `FlutterMap` widget as part of first frame** — confirmed via `router.dart:87-88` (`GoRoute(path: '/map', builder: (c, s) => const MapScreen())`) and `main.dart`'s `initialLocation: '/map'`. `MapScreen`'s own `initState` (`map_screen.dart:44-51`) is written defensively for exactly this concern (comment explains the map tab may start offstage inside an `IndexedStack`, and uses `Geolocator.getLastKnownPosition()` — described in-code as "~10 ms," OS-cached — as a fast-path before the live GPS stream takes over) — this is a genuinely thoughtful mitigation already in place, not a gap. **Severity: Low** for this specific sub-finding — the map being the first screen is a legitimate product decision (Phase 1 confirms it's the intended default), and the code already shows awareness of the cold-start cost a map widget can add, with a real mitigation. The residual cost is `flutter_map`'s own tile-layer initial paint (first tile fetch/render, an inherent cost of "show a map immediately," not something this codebase is doing wrong) — flagged for completeness, not as an actionable defect.

## 8. Battery Usage

Per the grounding brief: iOS background location tracking cannot actually engage (Phase 4/7, `NSLocationWhenInUseUsageDescription`-only, no path to "Always" authorization) — so the realistic battery assessment below is for **foreground-only continuous tracking**, which is what actually runs in practice on both platforms whenever the app is in the foreground and a truck is live.

- **Location settings: `LocationAccuracy.high`, `distanceFilter: 30` (meters), 10-second write-throttle** (`location_tracking_service.dart:31-46,52`). `LocationAccuracy.high` requests GPS-grade positioning (not the coarser/cheaper network/Wi-Fi-based tiers), and on iOS specifically it's paired with `ActivityType.automotiveNavigation` (`:43`) — the activity-type hint Apple's location subsystem uses to tune its own internal power/accuracy trade-off, appropriate for a food truck that's genuinely a moving vehicle, not a mis-set default. **Severity: Medium (inherent to the feature, not obviously mis-tuned, but worth being explicit about the real-world cost).** **User-facing impact (foreground-only, realistic case):** continuous high-accuracy GPS polling while an owner's app is open and their truck is live is one of the most battery-intensive things a phone can do — industry-standard estimates for continuous high-accuracy foreground GPS place drain in the **8-15% of battery per hour** range on typical modern hardware (this is a general mobile-platform figure, not something measured against this specific app/build in this pass — no on-device battery profiling tool was run). The 30 m distance-filter and 10-second write-throttle (`:52`) correctly reduce *network write* frequency for a stationary truck, but **do not reduce GPS radio/sensor polling frequency** — `distanceFilter` only gates which stream *emissions* get processed, the underlying OS location subsystem is still kept active at high-accuracy tier continuously while `start()` is running, per both `AndroidSettings`/`AppleSettings` configuration. **Recommended action:** for a food truck that's frequently stationary for hours at a single serving location, consider `LocationAccuracy.medium` (or a dynamic accuracy tier that only escalates to `.high` when recent readings indicate actual movement) as a real, meaningful battery win — the 30 m/10 s throttle already shows the team has thought about *write* cost; the same thinking hasn't yet been applied to *sensing* cost.
- **Android: `enableWakeLock: true` on the foreground-service notification config** (`location_tracking_service.dart:36`) — correct and expected for a genuine Android foreground service that must keep running while the screen is off (an owner's phone in their pocket while driving/serving) — this is the *right* tool for that specific job, not a battery-drain red flag by itself; flagged here only to confirm it was reviewed, not to recommend removing it (removing it would let Android throttle/kill the location updates the feature depends on).
- **iOS realistic behavior, given the broken background-permission path**: `allowBackgroundLocationUpdates: true` (`:45`) is set but per Phase 4/7 can never actually take effect without "Always" authorization, which this app's permission flow never requests. **Net effect: on iOS, tracking very likely stops (or is throttled to the OS's own low-power "significant location change" fallback, outside this app's control) the moment the app backgrounds** — meaning the *actual* iOS battery cost is bounded to foreground-only sessions, not true always-on background drain. This is a **better-than-advertised battery outcome purely by accident of the broken permission flow** — worth noting precisely because Phase 4/7 already correctly flag this as a functional bug (the feature doesn't work as designed); from a battery-harm perspective specifically, the same bug means iOS users are not silently bleeding battery in the background the way the code's own intent would otherwise cause. **Severity: Informational** (the functional-correctness severity is already captured in Phase 4/7; this note exists only to correct what the *battery* impact of that same bug actually is, per this audit's specific instruction to assess realistic runtime behavior).
- **No polling loops for real-time data** — confirmed clean. Order/booking/notification updates are pushed via Postgres Realtime channels and Firebase Cloud Messaging (`push_notification_service.dart`), not client-side polling. The only 2 `Timer.periodic` instances in the entire app are UI-only re-render ticks: `map_screen.dart:52` (1-minute timer to refresh the "opened Xd ago" pin-badge text) and `employee_dashboard_screen.dart:81` (1-minute timer to refresh a shift-duration display) — both are cheap `setState(() {})` calls with no network I/O, both correctly cancelled in `dispose()` (Phase 5 §2.9). **Clean finding** — no battery cost here beyond a once-a-minute empty rebuild on 2 screens.
- **No wakelock/keep-screen-on usage outside the Android location foreground-service notification** — `grep -rn -i "wakelock\|keepScreenOn\|WakelockPlus"` finds only the one legitimate location-service hit above; no screen is being kept artificially awake elsewhere in the app.

## 9. Memory

- **Riverpod family-provider realtime-channel accumulation (Phase 5 §2.7) — quantified here for a realistic session.** 19 non-`.autoDispose` family providers exist; the two most consequential for a **consumer** session are `foodTruckProvider(truckId)` (`food_truck_provider.dart:43-46`, opens a `'consumer-menu-$truckId'` Realtime channel per truck, per `:22-35`) and `truckReviewsProvider`/`myReviewProvider`/`truckFollowerCountProvider`/`announcementPrefProvider` (lighter-weight `FutureProvider.family` instances, cached but not channel-backed). For an **owner or employee** viewing bookings, `pendingBookingCountProvider` (`bookings_provider.dart:11-51`) opens a `'pending_bookings_$truckId'` channel per truck ID ever viewed. **Worst-case estimate for a single consumer browsing session**: a user who taps into 15-20 different truck profiles while exploring a city's food-truck scene over one sitting (a realistic, not adversarial, usage pattern for exactly the kind of "discover trucks near me" app this is) ends that session holding **15-20 permanently-open Postgres Realtime channels plus 15-20 retained `FoodTruck` model instances in memory**, none of which are released until the app process itself is killed — Riverpod's default (non-`.autoDispose`) behavior keeps every family-keyed instance alive for the provider container's lifetime once created, regardless of whether the originating widget is still mounted. **Severity: High** (restates Phase 5's finding with a concrete session-scale estimate). **User-facing impact:** individually each retained instance is small (a `FoodTruck` model + an idle Realtime WebSocket subscription), but the *channel* count matters more than the *memory* count here — Supabase Realtime connections are multiplexed over a shared WebSocket per client but each `.channel(...).subscribe()` still registers server-side topic subscriptions that count toward the project's Realtime concurrency, and 15-20 idle-but-subscribed channels per active user session, multiplied across concurrently active users, is a real, measurable server-side resource cost (cross-ref Phase 2 §6's realtime fan-out concern) as much as a client memory concern.
- **No image-cache tuning — Flutter's default `PaintingBinding.imageCache` limits (1000 images / ~100 MB) are left untouched** (`grep -rn "imageCache\|PaintingBinding" lib` → zero hits anywhere). Combined with §4's finding that most truck/menu images go through uncached `Image.network` (which still populates the *in-memory* `ImageCache`, just without a *disk* cache) and §6's finding that no `cacheWidth`/`cacheHeight` downsampling is applied anywhere: **every distinct image URL viewed in a session is held in memory at full decoded resolution**, up to the default 100 MB/1000-image cap. **Severity: Medium.** **User-facing impact estimate:** for a popular city with, say, 50-100 active trucks each with a logo + 3-5 menu-item photos + up to 10 photo-carousel images, a consumer scrolling through the map and tapping into several truck profiles in one session could plausibly decode 100-300+ distinct full-resolution images before Flutter's cache eviction kicks in — on a phone with a 3000px-long-edge source photo (§6), each single decoded bitmap can be tens of MB in raw pixel-buffer memory even though the compressed file is only ~1-2 MB, meaning the *decoded* memory cost is far larger than the *download* cost §4/§6 already flagged. This is the direct compounding effect of "no `cacheWidth`/`cacheHeight`" (this section) + "no upload-side resolution cap" (§6) + "no CDN transform" (§6) landing in the same place: real risk of visible jank or, on lower-memory Android devices, an OS-level low-memory image-cache eviction/OOM risk during heavy browsing, though no crash was reproduced or measured in this pass (this is a static-analysis-derived risk estimate, not an observed crash).
- **Unbounded-query in-memory footprint (cross-ref §5's 4 confirmed unbounded queries)**: for a *popular* city truck with a full year of order history, `orders_repository.dart:104` (`fetchOrdersForTruck`, no limit) loading into `order_queue_screen.dart`'s eager `ListView` (§2) could realistically hold **hundreds to low-thousands of `Order` model instances** (plus their embedded `order_items(*)` per Order, per the join at `orders_repository.dart:83`) in memory simultaneously at screen-open, all built as live widgets at once (§2's `ListView` vs `.builder` finding) rather than paged in. At today's data volume (`orders` table: 0 rows per Phase 2) this is entirely theoretical; the point of flagging it here is that the *code path* that will load an unbounded result set into memory already exists and is already wired to an eager-build list widget — the risk activates automatically as soon as real order volume accumulates, with no code change required to trigger it (i.e., it's a ticking-clock risk, not a hypothetical one contingent on some other bug).

## 10. Loading States

Building on Phase 6's design-lens finding (loading indicators are present on nearly every screen) — the performance-specific angle:

- **Every async boundary in the app is a blocking spinner; none are used to mask an operation that should instead be optimized, and none are skeleton/optimistic-UI.** `grep -rc "CircularProgressIndicator("` totals **55** across the app, `.when(...)` (the `AsyncValue` loading/data/error pattern) is used **22** times, and `grep -rn -i "shimmer\|skeleton"` returns **zero hits** in `lib/` and **zero** in `pubspec.yaml` (no `shimmer`/`skeletonizer`/similar package is even a dependency). **Severity: Medium.** **User-facing impact:** this means every one of the query-fan-out cases identified in §3 (truck profile's 5 independent providers, dashboard's 4) resolves behind separate, uncoordinated spinners rather than one shaped skeleton screen — the user sees content "pop in" in stages with visible layout shift between each arrival, rather than a smooth single transition from placeholder to content. This is a genuine "spinner compensating for un-batched queries" pattern per the audit brief's specific question — the fan-out in §3 is the root cause, and the all-spinner loading strategy is what makes that root cause visible/felt by the user rather than absorbed invisibly behind a single coordinated loading state.
- **No optimistic UI beyond the two specific cases Phase 5 already documented** (`setOpenStatus`/`updateOrdersAccepting` in `food_truck_provider.dart:115-160`, optimistic write + rollback-on-failure) — every other mutation (place order, respond to booking, submit review, toggle menu-item availability) shows a loading/disabled state and waits for the round-trip to complete before updating the UI, rather than assuming success and rolling back on failure. **Severity: Low** (a legitimate, conservative design choice for money-touching actions like orders/payments — optimistic UI is arguably *wrong* for a payment-confirmation flow — but worth noting explicitly since the brief asks whether the technique is used anywhere: it is, narrowly, only for the one non-financial toggle where the team already reasoned through the trade-off).
- **Loading states do not currently mask any N+1 query problem specifically** (this codebase's one confirmed N+1, `manage_hours_screen.dart:41-48`'s 7-call write loop per Phase 5 §2.15, is on the *save* path, which already shows a `_saving`-guarded spinner state — so the spinner there is arguably appropriately masking a real multi-call operation, not hiding an avoidable one, though the 7 calls should still be batched into 1 for the reasons Phase 5 already gives). The more accurate framing for this app is: **the spinners aren't hiding N+1 queries, they're hiding un-batched *fan-out* across separate features (§3)** — a related but distinct performance anti-pattern from classic N+1.

## 11. Client Responsiveness

- **No heavy synchronous work found that should move to `compute()`/an isolate — and no `compute()`/`Isolate` usage exists anywhere, which is appropriate given what's actually being processed.** `grep -rn "compute(\|Isolate\."` → zero hits. The only `jsonDecode`/`json.decode` calls in the entire app are in `places_autocomplete_field.dart:78,105`, decoding a Google Places Autocomplete API response — a small, bounded JSON payload (a handful of address suggestions), not a candidate for isolate offloading. Supabase's own client already deserializes query results internally; no large-payload manual JSON parsing exists on the main isolate anywhere else. **Clean finding** — this dimension is not a real problem given the app's actual workload shapes, and adding `compute()` here would be premature complexity, not a fix for anything.
- **Debounce is correctly present on both places it matters (map search, address autocomplete) and correctly absent from map camera movement (which needs a different fix, per §2, not a debounce)** — `map_screen.dart:142` (350ms, search text), `places_autocomplete_field.dart:63-64` (350ms, address autocomplete), both confirmed by Phase 5 §2.8 to cancel-and-restart correctly in `dispose()`. **The map-move handler (§2) is not "missing a debounce" so much as "doing full-rebuild work that shouldn't run per-frame at all, debounced or not"** — worth being precise that the fix there is memoization of the clustering math, with a debounce as a secondary/complementary mitigation, not the primary fix (a debounced-but-still-unmemoized rebuild would still redo the same expensive `.sort()`/clustering pass, just less often).
- **Double-submit / rapid-tap protection on payment and booking-response buttons is genuinely solid — a clean, positive finding.** Every payment-adjacent button checked disables itself during its own in-flight request: `order_cart_sheet.dart:162` (`(!hasItems || _paying) ? null : _pay`, cart checkout), `my_requests_screen.dart:719,730,759,780` (accept/decline/pay-deposit/pay-invoice, all gated on a single `_paying` flag), `send_estimate_sheet.dart:89`/`request_deposit_sheet.dart:115` (`_submitting` guard). **No double-charge risk was found on any payment button in this pass** — this directly answers the brief's ask to look at Phase 7's payment findings from a UX-double-charge angle: the *authorization*-layer issues Phase 2 found (client-trusted `amount_cents`, §Critical Finding #1 in `supabase-audit.md`) are real and severe, but they are not compounded by a client-side double-submit bug — a user cannot accidentally fire the same payment intent twice by rapid-tapping in this codebase. **One caveat worth flagging**: `my_requests_screen.dart` shares **one** `_paying` boolean across 4 different buttons (accept, decline, pay-deposit, pay-invoice) — functionally safe (no double-submit possible) but means tapping "Accept" also visually disables "Decline"/"Pay" simultaneously even though they're unrelated actions; a UX nit, not a correctness or performance bug, noted only because it sits in code adjacent to the double-submit question the brief asked about.
- **The one confirmed missing-`mounted`-check bugs from Phase 5 (§2.17, `my_orders_screen.dart:65`, `order_queue_screen.dart:99`, `calendar_screen.dart`'s second-`await` case) are responsiveness/crash-risk bugs, not performance bugs per se** — referenced here only to avoid re-litigating; not re-analyzed from a fresh angle since Phase 5 already fully characterized them and this audit found no additional instances beyond what Phase 5 already cited.

## 4. Prioritized Punch List (user-facing impact × effort-to-fix)

1. **Recompress `onboarding.png` (1.8 MB) and confirm/remove `icon.png` (1.6 MB) if unreferenced at runtime (§1/§6).** Zero code risk, no logic touched, immediately shrinks 87% of the asset bundle. Do this first — it's the cheapest win in the entire report.
2. **Add explicit timeouts to the Supabase/Edge-Function call layer (§3).** A hung request currently blocks the UI forever with no recovery path — this is the highest-severity *reliability-shaped* performance gap and the fix is centralizable (the same fire-and-forget wrapper Phase 5 §2.1 already flagged as duplicated 5× is a natural home for it).
3. **Fix the map screen's per-frame rebuild (§2).** Memoize `_applyClusterOffsets`/`_inVisibleBounds`/sort against the truck list + rounded viewport bounds instead of recomputing on every `mapEventStream` emission; this is also the root cause of Phase 6's live-observed stacked-pin bug, so this one fix closes a UX finding and a performance finding simultaneously.
4. **Add `maxWidth`/`maxHeight` to the 3 truck/menu-photo `ImagePicker` calls (§6)**, matching the pattern already correct in `account_screen.dart`'s avatar picker — small, mechanical, directly reduces upload size, download size (§4), and decoded memory footprint (§9) in one change.
5. **Replace the 17 `Image.network` call sites with `CachedNetworkImage` (§4)**, prioritizing map pins and truck-bottom-sheet images first (highest view-frequency). The package is already a dependency and already correctly used in 2 places — this is rollout completion, not new tooling.

7. **Add `.autoDispose` to the 19 family providers (Phase 5 §2.7, re-prioritized here for its memory/Realtime-connection angle, §9)** — same fix Phase 5 already recommends first; this audit adds the concrete estimate (15-20 permanently-open channels per heavy-browsing consumer session) as the performance-specific justification for prioritizing it.
