# Farlo — Code Quality Audit (Phase 5: Discovery)

**Scope:** `lib/` (116 Dart files, 26,793 lines), `test/`, `pubspec.yaml`, spot-checks into `supabase/functions/` where a client-side pattern has a server-side mirror. Discovery only — no code changes, no refactors, no `dart fix`. All paths relative to repo root `/Users/johnny/Desktop/Good Truck Finder`.

**Grounding:** Read in full before starting — `audit/architecture.md` (Phase 1) and skimmed `audit/supabase-audit.md` (Phase 2). This report does not re-litigate Phase 1's headline findings (18 direct-Supabase-call files, 6 god screens, near-zero tests, unused codegen deps) — it quantifies them exactly, finds every instance, and goes one level deeper into implementation-level issues Phase 1 didn't have scope to cover.

**Method:** `flutter analyze` (clean — 2 info-level lints, confirming lint-clean ≠ quality-clean, see below), plus ~60 systematic `grep`/`rg` passes across `lib/` cross-referenced by hand for every dimension below.

## 1. Executive Summary

**Overall verdict: functional but structurally fragile.** The app ships and `flutter analyze` is nearly silent, but that silence is misleading — the linter (`flutter_lints` default set, zero custom rules) doesn't catch any of the issues that matter for a codebase about to scale past one developer or past initial launch traffic: unbounded provider lifetimes, unbounded queries feeding non-lazy list widgets, 300+-line `build()` methods, and a test suite that is a single `expect(true, isTrue)`. Discipline is real in specific pockets (debounce on search inputs, `mounted` checks in ~156 of ~192 `setState` call sites, optimistic-update rollback, no circular provider graph, no N+1 loops beyond one case) — this is not a codebase that was written carelessly, it's one that was written fast under real deadline pressure (App Store rejections, live users) with quality gates that never got built.

**Top 5 highest-impact findings, across all dimensions:**

1. **18 `AsyncNotifierProvider.family`/`FutureProvider.family` instances and 1 realtime-channel-backed `StreamProvider.family` are not `.autoDispose`** (§7) — every truck/user/booking a person views during a session stays cached in memory for the lifetime of the app, and `pendingBookingCountProvider` (`bookings_provider.dart:11-51`) keeps an open Postgres Realtime channel + refetch cycle alive per truck ID forever. Only 8 call sites in the whole app use `.autoDispose`. This is the single largest state-management and memory-growth risk in the codebase.
2. **Four repositories issue fully unbounded queries** (`fetchOrdersForConsumer`, `fetchOrdersForTruck` in `orders_repository.dart:80,104`; `fetchOwnerRequests`, `fetchMyRequests` in `bookings_repository.dart:23,49`) with no `.limit()`, feeding straight into non-lazy `ListView(children: [...map(...)])` widgets in 4+ screens (§15) — a truck or user with a year of order/booking history will eventually load and render every row on every screen open.
3. **A 774-line dead file** (`lib/features/employees/widgets/shift_calendar_widget.dart`) and a 17-line dead file (`lib/core/widgets/loading_overlay.dart`) are never imported anywhere in the codebase — confirmed via cross-reference of every class declaration against every other file (§3).
4. **`_showAddDialog` employee-invite flow leaks a `TextEditingController`** and 15 empty `catch (_) {}` blocks plus 17 `debugPrint`-only catch blocks swallow errors with no user-facing surface — most critically `booking_chat_screen.dart:118-129`, where the message text field is cleared *before* the send call, so a failed send silently loses the user's typed message with zero feedback (§9, §16).
5. **`manage_hours_screen.dart:41-48` performs a 7-call sequential N+1 write loop** (one `upsertOperatingHours` round-trip per day of the week) instead of one batched call — non-atomic (partial-failure risk) and the only N+1 pattern found in an otherwise loop-free-of-awaits codebase (§15, §6).

Beyond these five: 63 raw `ScaffoldMessenger.showSnackBar` call sites with no shared helper (duplication), 48 raw hex `Color(0x...)` literals across 16 files despite a working `AppColors` design-system class existing — including re-declaring colors that are already named constants (§12), 45 raw route-string `context.go('/...')` calls with no `AppRoutes` constants class (§11), and 4 declared-but-zero-usage packages in `pubspec.yaml` (§4).

### 2.7 State Management

- **Headline finding — provider lifetime scoping is inconsistent and mostly wrong for a family-keyed cache.** Exhaustive count: **9 `AsyncNotifierProvider.family`** instances (`announcement_prefs_provider.dart:46`, `food_truck_provider.dart:44`, `employees_provider.dart:18,62`, `shifts_provider.dart:55,87,120,145,171`) and **9 `FutureProvider.family`** instances (`dashboard_screen.dart:44,60`, `bookings_provider.dart:58`, `favorites_provider.dart:79`, `map_provider.dart:47`, `reviews_provider.dart:11,16`, `planned_locations_provider.dart:13,24`) and **1 `StreamProvider.family`** (`bookings_provider.dart:12`, backed by a live Postgres Realtime channel) — **none of these 19 are `.autoDispose`.** Only **8 call sites total** use `.autoDispose` anywhere in the app (`dashboard_screen.dart:31`, `bookings_provider.dart:206,215,222`, `stripe_connect_screen.dart:15`, `transfer_provider.dart:26,71`, `employee_dashboard_screen.dart:27`). Practical effect: every distinct truck ID, booking ID, or user ID a person's session touches creates a permanently cached provider instance that Riverpod never tears down — for `pendingBookingCountProvider` specifically (`bookings_provider.dart:11-51`), that means an open Postgres Realtime channel (`Supabase.instance.client.channel('pending_bookings_$truckId')`, line 31) plus a live `refresh()` cycle stays subscribed forever per truck ID ever viewed, not just the currently-displayed one. **Severity: High** — this is a genuine, compounding memory/connection-count leak under normal usage (browsing multiple trucks, viewing multiple bookings in one session), not just under adversarial conditions.
- **No `ref.watch` misuse inside callbacks found** — systematic grep for `ref.watch(` appearing inside `onPressed:`/`onTap:` closures returned zero matches. **Clean, positive finding.**
- **No circular provider dependencies** (see §2.1).
- **Form state correctly kept local, not global** — spot-checked `edit_truck_screen.dart`, `manage_menu_screen.dart`, `register_owner_screen.dart`: all `TextEditingController`s are `State` fields, not provider state. No instance of ephemeral UI state incorrectly hoisted into a global provider was found.

### 2.8 Race Conditions (client-side only, per instructions — cron/webhook double-processing is Phase 3's territory, not repeated here)

- **`setOpenStatus` go-live/go-offline toggle has no in-flight guard against rapid double-tap.** `dashboard_screen.dart:132` wires a `Switch`'s `onChanged: (val) => _handleToggle(context, ref, val, truck.name)` directly to an unguarded async method (`dashboard_screen.dart:147-172` for the `isFixed` path) which itself calls `ownerTruckProvider.notifier.setOpenStatus` (`food_truck_provider.dart:115-124`). `setOpenStatus` reads `state.asData?.value`, optimistically writes new state, then awaits the network call with a catch-and-rollback (`food_truck_provider.dart:126-129`) — but there is no `_saving`/disabled-state guard on the `Switch` itself, so two rapid toggles fire two overlapping `setOpenStatus` calls against the same notifier state with no request sequencing. Worst case is a UI flicker/incorrect-final-state window, not data corruption (Supabase's own `update` is atomic per-call), but it is a real, reproducible race. **Severity: Medium.**
- **`manage_hours_screen.dart:36-52` `_save()` has a `_saving` guard on the button** (confirmed pattern is used correctly here) **but the 7-call sequential loop itself (§2.15) is non-atomic** — if call 3 of 7 fails, days 1-2 are persisted and days 4-7 are not, with no compensating rollback. This is as much a race/consistency issue as a scalability one. **Severity: Medium.**
- **Debounce is present and correct where it matters most**: `map_screen.dart:142` (350ms `Timer` on search-text-changed) and `places_autocomplete_field.dart:63-64` (350ms `Timer` on address autocomplete) both cancel-and-restart correctly in `dispose()`. **No missing debounce found on any search/autocomplete input** — positive finding.
- **Order/cart double-submit is correctly guarded** — `order_cart_sheet.dart:162` disables the pay button via `(!hasItems || _paying) ? null : _pay`. **Positive finding.**

### 2.12 Code Duplication

- **SnackBar-display boilerplate: 63 raw `ScaffoldMessenger.of(context).showSnackBar(...)` call sites across 24 files, zero shared helper.** Sampled 15 of the 63 (`dashboard_screen.dart:113,169,194,204,218,266,277,1383`; `manage_hours_screen.dart:53,64`; `subscription_screen.dart:85`; `background_location_disclosure.dart:38,54`) — the shape is near-identical every time (`SnackBar(content: Text(...), [backgroundColor], [duration])`), but error-path messaging is inconsistent in quality: some catches show a curated message (`'Could not update status: $e'`), others expose the raw exception text directly to the user (`'Error: $e'` at `manage_hours_screen.dart:65`) with no sanitization. A single `context.showError(String msg)` / `context.showSuccess(String msg)` extension would collapse all 63 sites and standardize the message-quality inconsistency in one place. **Severity: Medium** (duplication + inconsistent UX quality, high occurrence count).
- **Near-identical `_withStatus` record-copy helper duplicated verbatim (minus one field) between two notifiers**: `OwnerBookingRequestsNotifier._withStatus` (`bookings_provider.dart:126-147`) and `MyBookingRequestsNotifier._withStatus` (`bookings_provider.dart:175-196`) — both are 21-line methods that copy every field of a `BookingRequest` to produce a status-updated copy; they differ only in the `cancelledBy` value (`'owner'` vs `'consumer'`). This is exactly what a `BookingRequest.copyWith(status:, cancellationReason:, cancelledBy:)` method (which doesn't currently exist on the model, per Phase 1's note that no `copyWith` was audited) would eliminate. **Severity: Low-Medium.**
- **5× duplicated fire-and-forget async-IIFE wrapper pattern** — covered in §2.1.
- **Map tile URLs duplicated 3×** — covered in §2.11.
- **`_tosUrl`/`_privacyUrl` duplicated 2×** — covered in §2.11.
- **Raw hex colors re-declaring existing `AppColors` constants**: `Color(0xFF2563EB)` (== `AppColors.primary`) appears as a raw literal 2× outside `app_colors.dart` itself, `Color(0xFF6B7280)` (== `AppColors.textSecondary`) appears raw 4× outside the constants file. Total raw `Color(0x...)` literal count: **48 occurrences across 16 files**, including the amber "action needed" status color `Color(0xFFB45309)` repeated raw 5× (`booking_requests_screen.dart`, `my_requests_screen.dart`) despite `AppColors` existing and being actively imported in the same files for other colors. **Severity: Medium** — this isn't dead-simple "forgot the design system exists," it's "used the design system for some colors in this exact file and hand-typed hex for others," which is the more insidious version of the problem.

### 2.13 Complexity

- **Business logic (filtering/sorting) embedded directly in widget `build()`/`.when(data:)` callbacks, recomputed every rebuild with no memoization**: `booking_requests_screen.dart:205-224` filters+sorts the full request list inline inside the `data:` callback; `order_queue_screen.dart:135-153` and `my_orders_screen.dart:105-120` both do `orders.where(...).toList()` splits (active/history, incoming/in-progress/done) inline in the same place, on every provider emission. None of this logic lives in the notifier or a pure helper function that could be tested or memoized independently of the widget tree. **Severity: Medium** — not a performance emergency at current data volumes (ties back to §2.15's unbounded-query finding: as those lists grow, this cost grows too, compounding two separate issues in the same code paths).
- **Async/await consistency is genuinely good** — `Future.wait` is used **zero times** anywhere in the app (meaning there's no risk of the "one failure hides the others" `Future.wait` anti-pattern this audit was asked to check for, because the pattern simply isn't used at all) and no `.then()`-chaining was found anywhere in `lib/` (`grep -rn "\.then(" lib` — only the 1 legitimate `.then()`-free codebase-wide check confirms consistent async/await style, no callback-chain style mixed in). **This is a clean, positive finding** — but the flip side is zero use of `Future.wait` also means **no opportunity taken for concurrent independent fetches** anywhere (e.g., a screen needing both truck data and reviews data fetches them via two separate provider builds rather than one coordinated parallel fetch) — a missed-performance-opportunity note, not a bug. **Severity: Low.**
- **2 `void method() async` declarations** (bug-prone return-type pattern per dimension 17, cross-referenced here as a complexity/API-shape issue): `my_orders_screen.dart:65` and `order_queue_screen.dart:99`, both named `_openSheet(Order order) async`. Neither is awaited by its caller today (both are fired from `onTap:`), so the immediate bug risk is low, but the signature means no caller *could* await/catch them if a future change needed to — see §2.17 for the concrete downstream bug this pattern already causes (missing `mounted` check).

### 2.14 Testing

- **Confirmed: `test/widget_test.dart` is the only test file in the repository** (`find test -type f` → 1 file), containing exactly one test: `test('placeholder', () => expect(true, isTrue));`, with a stale comment ("Phase 1 focus: unit tests for auth logic") that was never acted on. **0% meaningful coverage.**
- **The codebase is not structured to be unit-testable without a live Supabase instance.** Every repository takes a concrete `SupabaseClient` in its constructor (not an interface — confirmed by Phase 1, `grep -rn "abstract class" lib` finds none in the repository layer) and every provider that needs data calls `Supabase.instance.client` or a repository built from it directly. There is no fake/mock `SupabaseClient` seam anywhere, and **no mocking library is even declared** (`grep -n "mockito\|mocktail" pubspec.yaml` → no matches) — confirming this isn't a "tests exist but coverage is thin" situation, it's "the tooling to write a unit test against business logic doesn't exist in the project yet."
- **No golden tests, no integration tests** (`integration_test/` directory does not exist).
- **Highest-value first tests to add (identified, not written, per audit scope)**:
  1. `AuthNotifier` state transitions (`auth_provider.dart:33-304`) — the most-depended-on provider in the app (§9 of Phase 1: "auth is a universal dependency"), currently has zero coverage of its timeout/rollback logic.
  2. Router `redirect` logic (`router.dart:44-73`) — pure function of auth/onboarding state, easily unit-testable without Supabase, currently governs every navigation in the app with zero tests.
  3. `OwnerTruckNotifier.setOpenStatus`/`updateOrdersAccepting` optimistic-update rollback (`food_truck_provider.dart:115-160`) — this is the one place the codebase has real state-machine complexity (optimistic write + rollback-on-failure) worth locking down before the §2.8 double-tap race is ever addressed, so a regression can be caught by a test rather than a user report.
  4. `OrdersRepository.placeOrder`/`bookings_repository.dart` quote/deposit flows — the two non-transactional multi-step writes Phase 1 already flagged; tests here would at minimum pin today's (partial-failure-prone) behavior before anyone attempts to fix it.

### 2.15 Scalability (code-level, not DB-level — Phase 2's territory)

- **4 unbounded repository queries, no `.limit()`, feeding non-lazy list widgets**: `OrdersRepository.fetchOrdersForConsumer` (`orders_repository.dart:80-89`), `fetchOrdersForTruck` (`:104-112`), `BookingsRepository.fetchOwnerRequests` (`bookings_repository.dart:23-30`), `fetchMyRequests` (`:49-57`) — all four `SELECT * ... ORDER BY ... ` with no row cap. Feeding into: **15 `ListView(children: [...])` (non-builder, eager) sites found** (`grep -c "ListView("` = 15 vs `ListView.builder(` = 2, `ListView.separated(` = 5) — of these, at minimum `dashboard_screen.dart:97`, `order_queue_screen.dart:140`, `my_orders_screen.dart:109`, and `booking_requests_screen.dart:214,747` were confirmed (by reading, not just grep) to `.map()` over the *full*, unbounded order/booking-request collection into eagerly-built widget lists — every row is built and laid out on every screen open regardless of scroll position, and there is no page-in/load-more mechanism at all. For a truck or user with a year of activity, this degrades from "seven months from now" to "materially slow today" for any owner running a busy truck. **Severity: High.**
- **No N+1 client-side query loops found except one**: `manage_hours_screen.dart:41-48` loops over `_entries.entries` (at most 7, one per day of week) issuing a sequential `await repo.upsertOperatingHours(...)` per iteration instead of one batched upsert call — 7 round-trips where 1 would do, and non-atomic (§2.8). Systematic search for `for (...) { ... await ... }` and `.map((...) async` patterns across all of `lib/` found no other instance — **this is a single, contained finding, not a widespread pattern.** **Severity: Medium.**
- **19 distinct Supabase Realtime channel-opening call sites** (`grep -c "\.channel('"` = 19, enumerated: `dashboard_screen.dart` ×3, `bookings_provider.dart`, `booking_chat_screen.dart`, `booking_requests_screen.dart`, `auth_provider.dart`, `map_repository.dart`, `food_truck_provider.dart` ×2, `truck_profile_screen.dart` ×2, `my_orders_screen.dart`, `order_queue_screen.dart`, `notifications_repository.dart`, `employees_provider.dart`, `employee_dashboard_screen.dart`, `employees_screen.dart`) — combined with the §2.7 finding that several of the provider-backed ones (`pendingBookingCountProvider` at minimum) never tear down because they're not `.autoDispose`, this is the same underlying risk Phase 1 flagged ("no code was seen capping concurrent channel counts") now with an exact call-site count and the specific mechanism (missing `autoDispose`) that makes it worse than "many channels" — it's "many channels that don't close." **Severity: High** (compounds directly with §2.7).

### 2.16 Error Handling

- **15 empty `catch (_) {}` blocks**, fully silent, no logging even in debug builds: `push_notification_service.dart:137`, `location_tracking_service.dart:73` (swallows reverse-geocoding failures — reasonable, address is optional), `dashboard_screen.dart:247`, `bookings_provider.dart:25`, `my_requests_screen.dart:690`, `auth_provider.dart:243,282,288,302` (4 of the 15 are RevenueCat `logOut`/`logIn` calls — arguably fine to swallow since they're best-effort sync-state calls), `map_repository.dart:31`, `map_screen.dart:61`, `notifications_repository.dart:67`, `employees_provider.dart:112,221`, `employee_dashboard_screen.dart:186`. Roughly half of these are defensible (best-effort background sync where failure is genuinely inconsequential) but roughly half are not narrated anywhere as intentional, meaning a future reader can't tell "this is deliberately swallowed" from "someone forgot to handle this." **Severity: Medium.**
- **17 `debugPrint`-only catch blocks** — a step up from silent, but still invisible to users and (since there's no crash-reporting SDK per Phase 1's note that Sentry/Firebase Crashlytics/etc. are entirely absent from `pubspec.yaml`) invisible in production too. Full list in §2.9's search already enumerated above; most concerning: `booking_chat_screen.dart:128` (send-message failure) and `booking_requests_screen.dart:647` (accept/decline-status-update failure) are **user-initiated actions with no error UI at all** — the user gets a stopped spinner and nothing else, no indication their action failed. Contrast with `subscription_screen.dart:73`, which does the right thing (`catch (e) { debugPrint(...); setState(() => _errorMessage = e.toString()); }`) — proving the pattern for surfacing errors exists in the codebase, it's just not applied consistently. **Severity: High** for the two user-action cases specifically (silent failure on a paid/committed action like a booking response is a real trust problem), **Medium** for the rest.
- **`booking_chat_screen.dart:118` clears the text input before the send call succeeds**: `_textController.clear()` runs at line 118, *then* `await _repo.sendMessage(...)` is attempted in the `try` block starting at line 120 — if `sendMessage` throws, the catch block at `:127-128` only `debugPrint`s, so **the user's typed message is gone from the input field and was never actually sent, with no error shown and no way to recover the text.** This is the single clearest error-handling bug found in this audit. **Severity: High.**
- **No inconsistent auth-vs-network error differentiation found** — spot-checked catch blocks across `auth_repository.dart`, `auth_provider.dart`: errors are generally caught as `catch (e)` and either rethrown or surfaced as `e.toString()`, with no evidence of a codebase-wide convention distinguishing `AuthException` from network/`PostgrestException` types for different user messaging — this is more an *absence* of a positive pattern than a broken one; there's no dedicated error-type-switch anywhere to audit as "inconsistent."

### 2.17 Async Issues

- **2 `void method() async` declarations** (§2.13): `my_orders_screen.dart:65-72`, `order_queue_screen.dart:99-106`. Both are `_openSheet(Order order) async { final result = await showModalBottomSheet<bool>(...); if (result == true) _load(); }` (or `ref.read(ownerOrdersProvider.notifier).load(...)`in the owner variant) — **neither checks `mounted` after the `await`**, and `_load()`/the `ref.read` call both touch `ref`, which throws if the `ConsumerState` has been disposed in the interim (e.g., user backgrounds the app or navigates away while the bottom sheet is open). This is a direct, reproducible instance of exactly the "await completes, then method continues to use `ref` without a mounted check" pattern the audit brief calls out. **Severity: Medium** (narrow timing window, but a real crash-shaped bug, not hypothetical).
- **Same gap, second occurrence, narrower window**: `calendar_screen.dart:113` correctly checks `if (type == null || !mounted) return;` after the *first* `await showModalBottomSheet`, but each of the 3 `switch` cases that follow (`:122-150`) does a **second** `await showModalBottomSheet(...)` and then calls `ref.invalidate(...)` immediately after, with **no repeated `mounted` check for this second await** — the same class of bug as above, just requiring two dismissals in sequence to trigger. Confirmed identical pattern duplicated in `shift_week_card.dart:109-149`. **Severity: Low-Medium** (narrower window than the two above, but same root cause, and duplicated across 2 files).
- **`Future.wait` used zero times anywhere** — per §2.13, this means the specific "one failure hides the others" `Future.wait` anti-pattern the brief asks about **cannot occur** in this codebase today, because the API is never called. Documented here as a explicit negative/clean finding rather than an absence-of-evidence gap.
- **Fire-and-forget is narrow and mostly intentional**: only 1 genuinely un-awaited, no-catch call was found: `employees_screen.dart:205-211`, `Supabase.instance.client.functions.invoke('send-employee-invite', ...).ignore()` — the explicit `.ignore()` signals intent (the author knows it's fire-and-forget and is suppressing the analyzer's unawaited-future lint deliberately) but there is **no catch/error handling at all** for this specific call, unlike every other `functions.invoke` in the app (20 total invoke call sites found; 19 of 20 are `await`ed with surrounding try/catch, only this one is bare `.ignore()`). If the employee-invite email fails to send, the owner is told "invited" (via the snack bar shown right after, `employees_screen.dart:212-218`, which does **not** depend on this call's outcome) with zero indication the notification email never went out. **Severity: Medium.**
- **No unnecessary `await` on non-Futures found**, and no missing `await` on a Future whose result matters (the 20 `functions.invoke` call sites and all mutation-repository methods were checked; all but the one `.ignore()` case above are properly awaited).

## 4. Prioritized Remediation List (discovery-framed — what to fix first and why)

1. **Add `.autoDispose` to the 19 family providers (§2.7), starting with `pendingBookingCountProvider`.** This is the highest-leverage single change in the report: it's a one-line-per-provider fix that directly closes the largest memory/connection-growth risk (§2.7) and materially reduces the realtime-channel-accumulation risk (§2.15) in the same motion. Do this before addressing anything else state-related.
2. **Add `.limit()` + convert the 4 unbounded query methods' consuming screens to `ListView.builder`/pagination (§2.15).** This is a launch-blocking-shaped risk, not a someday-nice-to-have — any truck or user with meaningful history will hit visibly degraded performance, and the fix (limit + builder) is small per call site even though there are ~9 total spots (4 repo methods + 5 confirmed screen call sites) to touch.
3. **Fix the `booking_chat_screen.dart:118` message-loss bug and the 2 confirmed missing-`mounted`-check bugs (§2.16, §2.17).** Three specific, cheap, high-confidence bug fixes (not a refactor) — clear the input only after confirmed send success; add `if (!mounted) return;` after the two `_openSheet` awaits.
4. **Delete the 2 dead files (§2.3)** — zero risk, immediate 791-line reduction, removes a trap for future contributors.
5. **Remove the 4 unused packages (§2.4)** — zero risk, `pubspec.yaml` cleanup, small build-time win.
6. **Establish one shared error/snackbar helper and route the 63 raw call sites through it (§2.12, §2.16)** — this single refactor simultaneously fixes the duplication finding and the inconsistent-error-message-quality finding (raw `$e` text shown to users in some places), and creates the natural home for finally adding the "silent failure on user action" fixes for `booking_requests_screen.dart:647` and other under-surfaced catches.
7. **Add named constants for the recurring magic numbers** (§2.10: durations, location-tracking throttle/distance values, page-limit values) and **an `AppRoutes`/`AppUrls` constants class** (§2.11) — low-risk, mechanical, and removes an entire class of future typo bugs in one pass, same spirit as the already-partially-adopted `SupabaseConstants`.
8. **Batch the `manage_hours_screen.dart` 7-call write loop into one call (§2.8, §2.15)** — small, contained, fixes both the non-atomicity race and the N+1 pattern at once; check whether a Postgres RPC/batch-upsert already exists server-side (Phase-2 territory) before writing new backend code.
9. **Start the testing foundation with the 4 identified highest-value targets (§2.14)** — this is the largest-effort item on the list and should be sequenced after the above tactical fixes, but it's the only way any of the fixes above 1-8 stay fixed rather than silently regressing; add `mocktail` (or similar) and target `router.dart` redirect logic first since it needs no Supabase seam at all.
10. **Extract sub-widgets from the 6 god screens, starting with `dashboard_screen.dart`'s 307-line `build()` (§2.2, §2.6).** Correctly sequenced last: it's the largest-effort, lowest-immediate-risk item, and doing it before items 1-3 would mean re-touching hundreds of now-relocated lines while the higher-value bugs are still live.
