### Critical Finding — closed this pass (the headline result of the re-verification)

`compile_user_data_export(uuid)` and `delete_account_data(uuid)` — both `SECURITY DEFINER` Postgres functions meant to be called only by their Edge Functions via the service-role key — were live-callable by **any unauthenticated caller** holding nothing but the public `anon` API key embedded in every client build. Confirmed exploitable with a real `curl` request against both the isolated branch and, briefly, production: `POST /rest/v1/rpc/compile_user_data_export {"p_user_id": "<any uuid>"}` returned that user's full profile, email, and account data with zero authentication. Root cause: `anon`/`authenticated` held explicit, direct `EXECUTE` grants that a first `REVOKE ... FROM PUBLIC` attempt didn't touch (caught by re-testing the live exploit after the first fix, not by assuming the migration succeeding meant the vulnerability was closed). Fixed by explicitly revoking from `anon`/`authenticated` and granting only to `service_role`; verified closed against both the branch and production via the same live exploit request (`HTTP 401` post-fix); permanent SQL regression test added (`security_abuse_scenarios.sql` scenario 12, 4 sub-cases, all passing). Full narrative in `REMEDIATION_LOG.md`, "Iteration 10 (continued) — full non-sampled re-verification pass: CRITICAL finding."
