## Architecture

Flutter + Riverpod 3.x + GoRouter (StatefulShellRoute — owner and consumer shells are separate indexed stacks). Supabase for auth/Postgres/RLS/realtime/storage/edge functions. Stripe Connect Express for payments (funds go direct to owner). FCM push via custom service-account JWT in edge functions. RevenueCat manages subscriptions (live on both iOS and Android). `business_type` ('mobile'|'fixed') drives GPS vs. stored address logic — fixed businesses show at their stored lat/lng, mobile businesses push GPS on open. **Farlo is a small business platform, not just food trucks** — design all features for any small food business type (pop-ups, brick and mortar, caterers, etc.).

**New this session:** `dashboard/` — a separate Vite + React + TypeScript app (not Flutter), client-side only, talking directly to Supabase with the same anon-key + RLS model as the main app. Founder-only via a new `is_founder()` email-check RLS helper. Deployed independently to Cloudflare Pages, not part of the Flutter build/release process.

## Recent Decisions

**Postgrest's `.order()` defaults to descending — a real bug hiding in 4 different tables, found while chasing a specials sort-order complaint (Jul 22):** Johnny reported "This Week's Specials" on the profile showing Friday before Tuesday. `fetchCurrentWeek()`'s query was `.order('event_date')` with no explicit direction — assumed ascending (the "obvious" default), which was wrong. Checked the actual installed package source (`postgrest-2.7.1/lib/src/postgrest_transform_builder.dart`) rather than keep guessing: `order(String column, {bool ascending = false, ...})` — this library's real default is **descending**. Grepped the whole codebase for every `.order(...)` call missing an explicit `ascending:` and found it wasn't just today's new code:
- `faq_repository.dart` (built *earlier this same evening*) — the FAQ had been showing categories and questions in reverse (Employees first, Getting Started last) since the moment it shipped.
- `planned_locations_repository.dart` (`fetchForMonth`/`fetchForWeek`) — **pre-existing**, predates this session entirely. The calendar's planned-locations ordering has likely been backwards since that feature was first built.
- `employees_repository.dart` (`fetchScheduledShiftsForMonth`/`fetchMyScheduledShiftsForMonth`) — same, pre-existing, month-view shift queries.
- `weekly_specials_repository.dart` — today's new code, same mistake, same assumption.

Fixed all four with explicit `ascending: true`. Deliberately left `employees_repository.dart`'s plain employee-list ordering (`invited_at`) alone — newest-invited-first (today's actual behavior) is a defensible, arguably intentional default for that view, not a clear bug, so it wasn't changed without being asked. **Lesson**: "ascending is the sane default" is an assumption, not a fact — check the actual library source once (or write one throwaway test) instead of trusting intuition about a query builder's defaults, especially for anything that predates the current session and could have been silently wrong for a long time with nobody noticing (sorted-but-reversed data often still "looks plausible" at a glance, which is exactly why this survived undetected in the calendar feature).

## Traps / Dead Ends

- **`_invalidateUserScopedProviders()` (called on sign-out) is a manually-maintained list, not something the type system enforces — a new realtime/stream provider that caches session-dependent data has to be added to it by hand, and nothing fails loudly if it's forgotten.** `activeTrucksProvider` (the map's live truck list) was missing from this list even though `foodTruckProvider` (the single-truck detail fetch) was already in it — so a truck only visible under a previous session's RLS context kept showing on the map after sign-out while its detail screen correctly came up empty. When adding any new `StreamProvider`/cached provider that reads anything gated by `auth.uid()` or RLS, add it to this list in the same PR — there's no compiler check that will catch a missed one, only a live symptom that looks exactly like a data/RLS bug rather than a caching one.
- **A cart/collection keyed by a single natural id (like `menuItemId`) silently breaks the moment two entries need to differ by something other than that id.** `CartNotifier`'s `Map<String, CartItem>` was keyed by `menuItemId` alone since, before menu customization existed, a given menu item could only ever appear as one cart line. Adding customization meant two different versions of the same dish (e.g. "no mustard" vs. "everything") would collide on that key and overwrite each other with no error — the second `add()` call would just silently replace/merge into the first. Fixed via a computed `cartKey` (menuItemId, or menuItemId+customization only when there is one) rather than assuming the natural id is always a valid unique key. When adding a variant/option to something that previously had exactly one instance per parent id, check every place that parent id is used as a map/collection key.
- **A consumer-facing Riverpod provider with realtime subscriptions for *some* tables on a row (e.g. menu items) but not the row's own table can make an owner-side toggle look completely broken to whoever's already viewing that data — with no error anywhere.** `foodTruckProvider` (drives `TruckProfileScreen`) had realtime for `menu_items`/`menu_categories` but not `food_trucks` itself — toggling `hours_hidden` (or anything else on that row) from the owner side never reached an already-open consumer profile screen. Compare against `ownerTruckProvider`, which already had the `food_trucks` subscription and worked fine — the fix was just adding the same pattern to the consumer-facing provider. When a "toggle X does nothing" report comes in for anything backed by a live-viewable screen, check for a realtime gap before assuming the toggle's own write logic is at fault.
- **Enabling a Google Cloud API (Places, Time Zone, etc.) under one project doesn't implicitly enable a *different* API in the same family** — reusing `GOOGLE_PLACES_API_KEY` for the Time Zone API returned a clean `REQUEST_DENIED`/`"This API is not activated on your API project"` even though Places itself works fine with the same key. Same category of gotcha as the RevenueCat/Play Developer API enablement earlier in this project — check Google Cloud Console → APIs & Services → Library for the *specific* API before assuming a working key covers a new endpoint in the same product family. **Resolved this instance had a second layer too**: this key also had API-level restrictions (Credentials → the key → "Select API restrictions") limiting it to only Places API — enabling Time Zone API project-wide wasn't enough on its own; it also had to be added to that key's specific allowlist before calls with it stopped 401ing.
- **`supabase functions deploy <name>` (no flag) silently resets `verify_jwt` back to its default of `true`, even for a function that was previously deployed with `--no-verify-jwt`.** There's no `config.toml` entry recording the per-function setting in this project, so the CLI has nothing to preserve it from — a routine redeploy (e.g. adding one field to an order payload) can break a DB-trigger-invoked function's auth with zero code change, surfacing as `{"code":"UNAUTHORIZED_INVALID_JWT_FORMAT"}` from the *platform gateway*, not the function's own `requireAgentSecret()` 401 response — a useful tell for telling the two failure modes apart. Bit `push-order-to-clover` twice in one session. **Always redeploy any agent/trigger-invoked function (anything using `requireAgentSecret`/the `agent_cron_bearer` pattern) with `--no-verify-jwt` explicitly, every time** — check `mcp__claude_ai_Supabase__get_edge_function` first if unsure what a function's current setting is before assuming a plain deploy is safe.
- **A clean, detail-free 401 (no scope error, no expiry message, identical across Bearer/Basic/query-param auth and sandbox/production) from Clover's API is a strong signal the *Merchant ID* is wrong, not the token.** Clover's real Merchant ID is alphanumeric (~13 chars, e.g. `4ZYA00K3H4JH1`) and is only reliably found in the URL of the merchant's own Clover web dashboard — a numeric ID that looks plausible (e.g. off a statement) is very likely a payment-processor MID instead, a completely different identifier that Clover's API will reject with zero explanation. Confirm the real Merchant ID from the dashboard URL before spending more time regenerating tokens.
- **Postgres cron jobs calling a non-existent function fail completely silently unless you check `cron.job_run_details` — an `EXISTS`-guarded trigger or a webhook-style cron with no caller watching its result will never surface the error any other way.** `extensions.http_post` does not exist in this project (only `net.http_post` does) — a cron job registered against it fails every single run with zero visible symptom. This exact bug was found in a brand-new cron job (`sync-truck-hours`) within 30 minutes of creating it, but checking `cron.job_run_details` for other jobs turned up the *same* bug on `check-open-businesses` (the "still open?" reminder), silently failing **100% of its runs (1083 failures)** since long before this session. Any time a cron-driven feature seems to "just not be happening," check `select * from cron.job_run_details where jobid = (select jobid from cron.job where jobname = '...') order by start_time desc limit 5;` before assuming the application logic is at fault.
- **Supabase Edge Functions run their own clock in UTC regardless of the business's real timezone — never compare `now()`/`new Date()` directly against a stored local wall-clock time (like `operating_hours.open_time`) without converting first.** Confirmed via a temporary debug field (`now.getHours()`, `now.getTimezoneOffset()`) that the runtime's local time genuinely is UTC — comparing that directly against Eastern-time hours entered by a real business is off by 4-5 hours depending on DST. Fix: derive local hour/minute/weekday via `Intl.DateTimeFormat('en-US', { timeZone: 'America/New_York', ... }).formatToParts(date)` (DST-aware) rather than any `Date` mutation method, which all operate in the server's own local time. Every business onboarded so far is in South Carolina — this is currently hardcoded to `America/New_York`, not a per-truck timezone column; revisit if a business outside Eastern time ever onboards.
- **A close time that crosses midnight (e.g. open `23:45`, close `00:00`) breaks a naive "minutes since midnight" comparison** — `00:00` is `0`, numerically *less than* any open time, so "is now before close" is always false and the business never closes. Detect `closeMinutes <= openMinutes` as an overnight window, add 24h to the close (and to "now" too, if "now" is still on the early-morning side of that same window) before comparing. Known remaining gap: only handles a window entered on a single day's `operating_hours` row — doesn't yet look back to *yesterday's* row for a window that started the previous day and is still active into today's early morning.
- **Android needs its own explicit Firebase Crashlytics Gradle plugin (`com.google.firebase.crashlytics`) and an explicit `firebase-bom` dependency — neither is automatic just from adding the `firebase_crashlytics`/`firebase_messaging` Flutter packages.** Without the Gradle plugin and a BoM forcing every FlutterFire plugin's transitively-pulled native Firebase library onto one consistent version, Firebase's reflection-based component discovery can fail to instantiate its own registrar classes (`NoSuchMethodException` on `XyzRegistrar.<init>`, symptom: `NullPointerException: FirebaseCrashlytics component is not present` thrown inside `Firebase.initializeApp()`) — a hard crash on every single app launch, in every release build, with no way to recover past the splash screen. Found via a real Google Play rejection ("does not open or load") on Jul 16, likely present since Crashlytics was first added, since nobody had tested a real Android release build's actual launch before. - **A long-running Android emulator instance's virtual network can silently degrade until it's indistinguishable from a real app bug — a fresh boot is the fastest way to rule this out.** After ~1 hour of continuous uptime, an emulator instance started failing every external DNS lookup (`Failed host lookup`) for multiple unrelated hostnames simultaneously (map tiles, RevenueCat, an unrelated analytics SDK) — a strong tell it's the emulator's environment, not app code, since a real code bug wouldn't uniformly break every third-party network call at once. `adb shell ping` is not a reliable way to check this either way — it returned obviously-corrupted output (impossible multi-quadrillion-millisecond timings, wrong data bytes) on both the degraded instance *and* a completely fresh boot, suggesting `ping`'s ICMP timing is just broken under this emulator image regardless of real network health. Launching the same command with the tool's own `run_in_background: true` (no trailing `&`) kept it alive correctly across subsequent tool calls.
